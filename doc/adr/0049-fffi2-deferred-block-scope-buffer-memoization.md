# ADR-0049: Per-call-site buffer-size memoization for FFFI2 deferred-block scopes

## Context

### What `DeferredBlockScope` is and why it allocates

[`src/go/public/thestack/fffi2/runtime/fffi2_rt_deferred.go`](../../src/go/public/thestack/fffi2/runtime/fffi2_rt_deferred.go) implements the per-frame capture buffer that backs every widget primitive whose body is emitted out of order from its header â€” `c.EndETable` (cells + headers), `c.DockArea` (per-tab bodies), `c.Window`/`c.Tooltip` (tip + target), `c.NodeDir` (node body), and so on. Each `DeferredBlockScope` owns two `*bytes.Buffer`:

- `dataBuf` â€” slab holding all block keys plus the captured opcode payload, constructed via `bytes.NewBuffer(make([]byte, 0, 64*1024))` (`fffi2_rt_deferred.go:77`)
- `tempBuf` â€” capture target redirected from `Fffi2.SendIntermediate` between `Begin()` / `End()`, constructed via `bytes.NewBuffer(make([]byte, 0, 4096))` (`fffi2_rt_deferred.go:78`)

Construction is generated by `fffi2/compiletime/goserver/fffi2_compiletime_go_server.go:624` into `bindings/factories.out.go` â€” one scope per `deferred*` field on every Fluid builder type, instantiated **per widget call**. The Fluid is short-lived: it lives from `c.X(...)` until `et.Send()`, where `SpliceDeferredBlockMap(scope)` copies the captured bytes into the `RetainedFffiBuilder`'s buffer and the scope falls out of reference. The OS pipe handles writing the assembled frame to the Rust client; the scope itself becomes GC garbage.

### What pprof says about the cost

The remaining 41 % through `WriteToFixedKey` grows the *consumer* `RetainedFffiBuilder.buf`, which is pooled separately at `fffi2/typed/fffi2_typed_impl.go:48` but with a 4 KiB cap (`largestPooledBuffer`) that excludes virtually every real-world buffer from reuse â€” a related issue, but out of scope here.

### Why we measure cost too, not just allocate

The original presenting symptom was a visible imztop stutter. After fixing the proc-collector heap pressure (35 GB â†’ 372 MB per 30 s in `proc.(*Collector).collectAll`, see the scratch-buffer refactor in [`src/go/public/observability/sysmetrics/proc/proc.go`](../../src/go/public/observability/sysmetrics/proc/proc.go)), the visible stutter persisted. Slow-frame instrumentation at [`src/go/public/thestack/imzero2/metrics/metrics.go`](../../src/go/public/thestack/imzero2/metrics/metrics.go) showed `sync_us` â‰ˆ 25-32 ms on 5-10 % of frames, with `written_b` â‰ˆ 250-330 KB per frame consistently across all frames (slow and fast). The stutter source turned out to be Rust paint occasionally overrunning the 16.6 ms vsync deadline â€” not heap pressure on the Go side.

That diagnosis taught us something the existing metrics don't surface: **per-widget-primitive wire-byte cost is not directly observable today**. We had to infer it by elimination from the slow-frame logger and grep through alloc profiles. A measurement primitive that reports "the cells scope cost 280 KB this frame; the tooltip target cost 0.5 KB" would have shortcut the entire investigation.

### Forces

- **Allocation pressure is real but not the dominant stutter cause.** Cutting `growSlice` is worth doing, but the design choice should not be ranked primarily on raw allocation bytes saved. Both candidate solutions kill `growSlice` for the addressed surface.
- **Observability of widget cost has zero existing surface.** Frame-level wire bytes are recorded (`metrics.FrameMetrics.LastWritten`); per-primitive bytes are not. Adding instrumentation in widget code is a slippery slope â€” it scales with widget count, requires constant discipline, and creates a second debug-only code path.
- **`sync.Pool` carries operational hazards.** Go #23199 (oversize buffers pinned indefinitely), per-P shard quirks, victim-cache GC semantics, and the Release-correctness invariant (must put back exactly once, must not touch after Put). All known, all manageable, all extra cognitive load.
- **Codegen is already the seam.** `fffi2/compiletime/goserver/fffi2_compiletime_go_server.go` emits the scope-construction line and the Send-method body that consumes the scope. Whatever sizing-aware behavior we introduce must be expressible as one extra emitter slot.
- **The build must stay CGO-free** (ADR-0026 invariant) and must not introduce a mandatory new codegen step beyond `./generate.sh`.

## Design space (QOC)

**Question.** How should `DeferredBlockScope` allocate its `dataBuf` and `tempBuf` to (a) eliminate the per-frame `bytes.growSlice` traffic and (b) make the per-primitive wire cost observable, without taking on hazards disproportionate to the wins?

**Options.**

- **O1 â€” sync.Pool the buffers.** Pool `*bytes.Buffer` instances; `NewDeferredBlockScope` `Get()`s, codegen emits `Release()` calls after `SpliceDeferredBlockMap` that `Put()` them back. Apply a per-pool oversize cap to mitigate Go #23199.
- **O2 â€” Per-call-site memoization (this ADR).** Each scope kind (`cells`, `headers`, `tabBody`, `nodeBody`, `tip`, `target`, `rows`) gets an `atomic.Uint64` hint. `NewDeferredBlockScope` allocates `make([]byte, 0, hint)`; on Release the scope folds its observed high-water mark back into the hint via a CAS loop with slow exponential decay. Side effect: the hint table is a permanent zero-cost cost-attribution surface available to overlays, tests, and HTTP endpoints.
- **O3 â€” Hybrid: pool + memoize.** Buffers come from a per-kind pool keyed on size class; misses fall back to a memoized initial allocation. Pool hits give zero-alloc reuse; misses give first-try right-sized allocation.
- **O4 â€” Do nothing.** Accept the 180 MB/s scope-buffer churn; it's not the visible stutter source.

**Criteria.**

- **C1 â€” Allocation traffic reduction** (measured against the 5.4 GB / 30 s scope-internal `growSlice` budget).
- **C2 â€” Observability bonus.** Does the design produce a re-usable measurement of per-widget-primitive wire cost?
- **C3 â€” Operational simplicity.** Number of failure modes, depth of test surface, behavior under panics, behavior under one-off oversize frames.
- **C4 â€” Codegen-template footprint.** How many emitter slots in `fffi2_compiletime_go_server.go` change, how invasive the change to `methods.out.go`.
- **C5 â€” Compatibility with the future `RetainedFffiBuilder.buf` pooling work.** Whether this design composes with or blocks a follow-up that fixes the 41 % `WriteToFixedKey` share.

|    | O1 (sync.Pool) | O2 (memoize) | O3 (hybrid) | O4 (do nothing) |
|----|----|----|----|----|
| **C1** allocation savings | ++ (eliminates base alloc too) | + (eliminates growth; base alloc remains) | ++ (best of both) | âˆ’âˆ’ |
| **C2** observability bonus | âˆ’âˆ’ (none) | ++ (per-kind cost gauge) | + (memoized hint still exists in the hybrid) | âˆ’âˆ’ |
| **C3** operational simplicity | âˆ’ (oversize cap, Release-correctness, per-P quirks) | + (one atomic CAS, no pool to misuse) | âˆ’âˆ’ (every hazard of O1, plus a tunable interaction with the hint) | ++ |
| **C4** codegen footprint | + (one new emitter slot) | + (one new emitter slot + hint var per kind) | âˆ’ (two emitter slots, two storages) | ++ |
| **C5** composition w/ future builder-pool work | + (orthogonal) | ++ (hint feeds builder-pool too) | + (orthogonal) | n/a |

The dominating axis is **C2**. O1 and O4 produce no measurement surface; O3's measurement comes from the same memoization layer O2 introduces. O2 leaves ~80-100 MB/s on the table compared to O1 (the per-scope base allocation), but recovers all of it in surface area: the hint table is an observability primitive future work can build on (frame-cost histograms, per-kind budgets, CI regression checks, adaptive UI), where the pool produces only the allocation reduction.

## Decision

We will introduce **per-call-site memoization of buffer-size hints** as the sizing strategy for `DeferredBlockScope.dataBuf` and `DeferredBlockScope.tempBuf`, and treat the hint table as a first-class observability surface.

Concretely:

1. **One `atomic.Uint64` per scope kind**, declared in a generated companion file. Scope kinds are taken from the IDL `deferredBlockMap` names that already exist in `fffi2/compiletime/goserver/fffi2_compiletime_go_server.go:624`'s emitter (`cells`, `headers`, `tabBody`, `nodeBody`, `tip`, `target`, `rows`, â€¦):

   ```go
   // src/go/public/thestack/imzero2/egui2/bindings/deferredhints.out.go â€” generated
   package bindings

   import "sync/atomic"

   var (
       hintCells       atomic.Uint64
       hintHeaders     atomic.Uint64
       hintTabBody     atomic.Uint64
       hintNodeBody    atomic.Uint64
       hintTip         atomic.Uint64
       hintTarget      atomic.Uint64
       hintRows        atomic.Uint64
   )
   ```

2. **`NewDeferredBlockScopeHinted` constructor variant** in `src/go/public/thestack/fffi2/runtime/fffi2_rt_deferred.go` that takes a `dataHint *atomic.Uint64` and pre-sizes `dataBuf` from it (with a floor â€” `deferredDataBufFloor = 4 KiB` â€” to keep tiny scopes from triggering re-grow on their first byte). The legacy `NewDeferredBlockScope` remains as a thin shim wrapping a process-static hint, for non-codegen callers.

3. **`(*DeferredBlockScope).ReleaseWithHint()` method** that folds the observed `dataBuf.Len()` back into the hint via a CAS loop with peak-and-slow-decay update rule:

   ```
   if observed >= old: next = observed                          // ratchet up
   else:               next = old - ((old - observed) >> 5)     // 1/32 step toward observed
   ```

   The decay shift (5) gives a ~30-frame half-life â€” fast enough to recover from a transient spike within ~half a second at 60 Hz, slow enough that frame-to-frame jitter does not destabilise the hint into oscillation. The Release also drops references to `dataBuf`, `tempBuf`, `entries`, and `getFffi` so the buffers GC.

4. **Codegen emits the `ReleaseWithHint()` call** after the last `SpliceDeferredBlockMap` for each scope in every generated `Send()` method. The natural emitter slot is the same one that emits the existing `r.SpliceDeferredBlockMap(inst.deferredCells)` line at `methods.out.go:1220` and siblings.

5. **`ScopeHintsSnapshot()` API** in `src/go/public/thestack/fffi2/runtime/` returning a `[]ScopeHintSnapshot` (kind name, observed high-water bytes). Wire it into:
   - **`/debug/pprof/custom/scopehints`** HTTP handler under the existing pprof listener (`--pprofHttpListenAddress`).
   - **The slow-frame logger** at `src/go/public/thestack/imzero2/metrics/metrics.go`'s `RecordBytes`: when a frame exceeds `SlowFrameThresholdNs`, log the top-N scope kinds by current hint alongside the existing render/sync/interpret/written/read breakdown.
   - **The metricsoverlay widget** (`src/go/public/thestack/imzero2/egui2/widgets/metricsoverlay/metricsoverlay.go`) as an opt-in column behind an env-toggle, so the dev demo can read per-kind cost without rebuilding.

Codegen change footprint, in lines, against `fffi2_compiletime_go_server.go`:

- One new function emitting `deferredhints.out.go` (declares the per-kind atomic vars). Run once at codegen time, indexed by the IDL `deferredBlockMap` names.
- One modification to the per-scope construction line at line 624 to thread the hint pointer.
- One modification to the per-Send-method emitter to append `inst.deferredX.ReleaseWithHint()` after each `r.SpliceDeferredBlockMap(inst.deferredX)`.

## Alternatives

- **O1 â€” sync.Pool the buffers.** Rejected primarily on **C2 (observability bonus)** rather than on its allocation merits, which are strictly stronger than O2 by ~80-100 MB/s on the base-buffer path. A pool produces only allocation savings; this ADR's design produces allocation savings *and* a permanent passive cost-attribution surface that future work (CI regression tests, frame-cost overlays, adaptive UI, payload-budget planning) can build on without re-instrumentation. O1's operational hazards (oversize cap, Release-correctness invariant, sync.Pool's per-P + victim-cache GC semantics) compound the design cost. If the post-implementation measurement shows the base-buffer allocation is still load-bearing, we revisit by layering a pool *under* the memoization (the hybrid in O3) as a Tier-2 ADR update, not a supersession.
- **O3 â€” Hybrid: pool + memoize.** Rejected as premature. It inherits every hazard of O1 while only marginally improving on O2's allocation savings. Reserve as a follow-up after the memoization design is proven in steady-state.
- **O4 â€” Do nothing.** Rejected because the 180 MB/s of `growSlice` is the second-largest allocator in the system after the (already-fixed) proc collector, and the design's actual headline value â€” observability â€” is not available from any alternative path without a separate instrumentation effort.

## Consequences

### Positive

- **`bytes.growSlice` traffic from `DeferredBlockScope` drops to zero in steady state** after the hint stabilises (~30 frames per kind under the 1/32 decay shift). At 5.4 GB / 30 s budget that's ~180 MB/s removed from `growSlice` cum. The base `make([]byte, 0, hint)` cost â€” one per scope per frame â€” remains; with hint stability this is roughly the same total bytes as the eventual buffer would have grown to, but allocated in one shot rather than as a series of doublings.
- **Per-widget-primitive wire cost becomes a first-class, passive observable.** No application code changes are needed to gain visibility into which FFFI2 primitive contributes how much to a frame's wire payload. The hint table is updated as a side effect of sizing; observability does not cost extra CPU at steady state beyond the one CAS per Release that the sizing requires anyway.
- **The slow-frame logger gains attribution.** A future log line for a slow frame can include "top contributors: cells=262144 tabBody=15234 rows=8192" alongside the existing breakdown, so triage moves from "Go side or Rust side?" to "*which* Go-side scope drove this frame's payload?" in one step.
- **CI gains a regression bar at no extra runtime cost.** Tests can snapshot a hint at the end of a deterministic scene render and assert it against a budget; a PR that doubles a widget's per-row cost fails at test time, not in production.
- **The decision composes with the future `RetainedFffiBuilder.buf` pool work.** A pooled builder buffer can pre-size from `sum(hints)` at the start of a frame and avoid its own growth chain. The hint table becomes a planner input.

### Negative

- **The base `make([]byte, 0, hint)` cost is not amortised across frames.** Compared to a pool, each scope still allocates one buffer per frame; only the doublings disappear. For a hint-stable scope this is roughly 1 alloc per scope per frame at exactly the right size, which is acceptable. If `sum(hints) Ã— scopes_per_frame Ã— fps` ever climbs above the GC trigger threshold meaningfully, the pool-as-Tier-2-update path is open.
- **Per-kind aggregation, not per-instance.** Two `c.EndETable` instances on screen share `hintCells`; the hint reports the max across all simultaneous cells-using widgets. Fine for budgeting and triage; insufficient for "which specific ETable instance is expensive?". Recoverable by widening the key to `(kind, widget_id)` with a `sync.Map`, which is a larger change deferred to a follow-up ADR.
- **Hint oscillation under pathological workloads.** The 1/32 decay shift is empirical. A workload that toggles a scope's size in a sawtooth pattern faster than the half-life will see the hint hover near the upper envelope, over-allocating relative to the median frame. The decay shift is tunable; if observed in production we move it to a build-tag or env knob rather than re-litigating in code.
- **One additional emitter slot in `fffi2_compiletime_go_server.go`.** Manageable in isolation; cumulative codegen complexity needs a generator-level explanation pass eventually (out of scope here).

### Neutral

- **Replaces an implicit assumption with an explicit hint.** Before: each scope's `64*1024` initial cap is hard-coded at `fffi2_rt_deferred.go:77`. After: the cap is a per-kind hint with an explicit update rule. Tunability moves from "edit the constant" to "tune the decay shift or the floor". Net code complexity is comparable; the locus of behavior changes.
- **The hint table grows with the number of IDL deferred-block kinds.** Each new IDL deferred block adds one `atomic.Uint64` (8 bytes) and one `ScopeHintSnapshot` entry. Negligible.
- **Process restart resets all hints to zero.** The first frame after start allocates from the floor (4 KiB) and grows on the cold path; the cost of that first frame's growth is bounded and not perceptibly different from cold-start under the current design.

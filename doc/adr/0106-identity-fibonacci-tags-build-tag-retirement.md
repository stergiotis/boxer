# ADR-0106: identity — fibonacci-coded tags become the only scheme; the identifier build-tag axis is retired

## Context

`public/identity/identifier` currently implements **fixed-width** tagged
64-bit identifiers: a compile-time tag width selected by one of eight build
tags (`identifier_tag_fixed4` … `identifier_tag_fixed32`), with the active
width pinned repo-wide in `./tags` (currently 16). The sibling packages
`fibonacci` and `fibonaccicode` carry the arithmetic for the **variable-width**
scheme the tree was originally designed around — a tag encoded as the
fibonacci code of the tag value, MSB-aligned, self-delimited by its trailing
`11` comma — but no identifier implementation uses them: in this repository
the variable-width scheme exists only as support math, consumed by one CLI
advisor command and one compression experiment.

Two pressures force a decision now.

1. **A package review (2026-07-05) confirmed nine defects by execution.**
   They cluster exactly on the seam between the two schemes and in the
   untested support code: an off-by-one in `fibonacci/stats.go`'s tag-value
   class bounds (visible in the `lw id tagvalue leadingzero` output, which
   silently omits the largest tag value of every width class), runtime panics
   on degenerate inputs in `fibonaccicode` and `fibonacci`, a too-weak
   compose-time guard that lets an oversized body silently corrupt the tag,
   an unhandled transaction-size limit in the Badger batch interner, and a
   stable-id violation when two generators serve the same tag concurrently.
   Two further packages (`identcontainer`, `vlq`) have no consumers at all.

2. **The historical blocker for variable-width tags is gone.** The reason the
   fixed-width scheme survived was operational: ClickHouse-side SQL could not
   cheaply split a variable-width id into tag and body. This is disproven
   empirically. A pure-SQL decoder using only integer built-ins
   (`bitAnd`/`bitShiftLeft`/`bitOr`/`bitShiftRight`/`bitCount`/`bitTest`/
   `arraySum`/`range`) was validated on ClickHouse 26.6 against 2,817 golden
   ids produced by this repository's own Go encoder — every width transition
   from 2 to 12, every fibonacci boundary ±1 up to the uint32 maximum, and
   extreme and random bodies — with zero mismatches on tag width, tag value,
   and body.

The fixed-width scheme's remaining advantages are a one-instruction split and
a uniform body size. Against that, fibonacci-coded tags are self-describing
(no out-of-band width constant needed to split an id — in any language),
frequency-adaptive (hot tags take short codes and wide bodies; rare tags take
long codes), and remove the *only* reason the `identifier_tag_fixed*` build
tags exist: baking one global width into the binary.

## Decision

### SD1 — one scheme: MSB-aligned fibonacci-coded tags

A `TaggedId` is `tagBits | body`. The tag bits are the fibonacci code of
`tagValue − 1` (Zeckendorf representation, reversed, terminated by the `11`
comma), placed MSB-aligned; the body occupies the remaining low bits. Tag
value `0` is unencodable and therefore invalid — this aligns with the
existing invariant that body `0` is reserved as invalid/NULL per tag. The
fixed-width scheme is removed, not kept alongside (see Alternatives).

Two properties carry the design and MUST be pinned by tests:

- **Self-delimitation under adversarial bodies.** Scanning from the MSB, the
  first adjacent `11` pair is always the tag's comma: the Zeckendorf part
  contains no `11`, and any pair contributed by body bits sits strictly
  below the comma. Splitting therefore needs no external width.
- **Prefix-freedom ⇒ cross-tag uniqueness.** No tag's code is a prefix of
  another's (a shorter code ends in `11`; a longer code has no `11` that
  early), so two distinct tags can never produce the same 64-bit id.

### SD2 — one split algorithm, two golden-locked implementations

The split is specified once and implemented twice (Go and SQL); a golden
corpus generated by the Go encoder MUST be decoded bit-identically by the
SQL implementation in CI (clickhouse-local server-truth harness):

```
pairs    = id AND (id << 1)            -- upper bit of every adjacent 11-pair
smeared  = pairs OR pairs>>1 OR pairs>>2 OR pairs>>4 OR pairs>>8 OR pairs>>16 OR pairs>>32
width    = 66 − popcount(smeared)      -- tag width incl. trailing comma bit
bodyMask = (1 << (popcount(smeared) − 2)) − 1
body     = id AND bodyMask
tagBits  = id AND NOT bodyMask
tagValue = Σ_j fib(j+2) · bit(id, 63−j)   for j in [0, width−2]
```

`pairs = 0` means "no comma": the id is malformed and both implementations
MUST report it as invalid rather than return a value. Note the encoder's two
biases cancel: the Zeckendorf sum of the tag bits equals the tag value
directly — no ±1 adjustment exists anywhere in the decoder.

For a **known** tag the split degenerates to the same mask-and-compare as
fixed-width: `IdTag` remains a bare `uint64` bit pattern, and its width, tag
value, body mask and maximum body are derived (and cacheable) from the bits
alone. The ~12-ALU-op general split is paid only when the tag is unknown.

### SD3 — the identifier build-tag axis is retired

The eight `ident_tw*.go` stubs, `ident_impl_const_tw.go`'s compile-time
constants (`TagWidth`, `MinTagWidth`, `MaxTagWidth`, and the misleading
`Uint32TagValueTagWidth = 45` — the measured maximum is 47), and the
`identifier_tag_fixed16` entry in `./tags` are deleted. Editor/CI guidance in
ENGINEERING_PRACTICES §3 is updated; the historical mention in ADR-0004
stays as written. This follows the same playbook as the `llm_generated`
build-tag retirement (ADR-0083): do not reintroduce scheme-selection tags.

### SD4 — tag-value domain stays `uint32`; planning API replaces the stats table

`TagValue` remains `uint32`. Measured consequence: code widths span 2–47
bits, so every tag retains a body of at least 17 bits (≈131k ids) and small
tag values get up to 62 bits. `fibonacci/stats.go` — the precomputed
`StatsHolder` with the off-by-one class bounds, the never-populated example
column and the commented-out checks — is deleted. `SelectFittingTagValueRange`
(whose bounds are correct) becomes the core of a small **tag planning** API:
pick a tag-value range from expected id cardinality, optionally preferring
values whose codes carry leading zeros (column-compression-friendly). The
`lw id tagvalue leadingzero` CLI advisor is rebuilt on it.

### SD5 — ClickHouse surface: nanopass macros plus emitted UDFs

The decoder ships as a nanopass macro family in canonical function-call form
— `lwIdTagBits(id)`, `lwIdBody(id)`, `lwIdTagWidth(id)`, `lwIdTagValue(id)`,
and `lwIdHasTag(id, tagValue)` — expanded by the SQL pipeline into the SD2
expressions. `lwIdHasTag` expands to the sargable range form
(`id BETWEEN tagBits AND tagBits OR bodyMask`), preserving primary-index
pruning; tag-equality filtering never pays the general split. In addition,
the leeway DDL stage can emit equivalent `CREATE FUNCTION` statements so
hand-written SQL uses the same names. An id→name dictionary fed from the tag
registry is **deferred** until a consumer needs names rather than values.

### SD6 — identgen hardening folds into the port

The `identifier.IdGeneratorI` seam is unchanged for consumers. While porting
the backends to per-tag body capacity:

- **Per-tag exclusivity at the factory.** `Create` for a tag with a live
  generator on the same store errors; the per-instance mutex demonstrably
  fails to protect the get-or-assign window across instances (two ids were
  minted for one key under contention).
- **Batch writes survive the transaction-size limit.** The Badger batch
  interner commits fresh mappings in as many transactions as needed
  (commit-and-continue on `ErrTxnTooBig`; a 500k-key batch currently fails
  outright). This is safe because interning is idempotent get-or-assign: a
  persisted prefix of an errored batch simply resolves as existing on retry.
  The "one write transaction" wording in the batch contract is corrected.
- **Backend parity near exhaustion.** The in-memory interner drops its
  worst-case up-front capacity check in favour of the Badger backend's lazy
  behaviour, so the two backends accept the same batches.
- The sequential generator stops logging a warning per call when a natural
  key is supplied; the contract ("the key is ignored") moves to the seam's
  documentation.
- Exhaustion becomes testable without injection seams: a long tag (e.g.
  60-bit code ⇒ 15-id body) exhausts naturally in a unit test.

Badger stores need **no data migration**: they persist bodies and tag values
only (mapping keys are prefixed by the tag value, the stored value is the
body), never the composed bit layout.

### SD7 — scope cull

`identcontainer` (roaring64 set and slice wrappers, zero consumers, including
the 32-bit interface vocabulary with no implementation) and `vlq` (encode-only,
zero consumers) are **deleted**. Both are recoverable from git history; either
returns only with a concrete consumer.

### SD8 — clean break, no dual decode

No fixed-width fallback decoder is shipped. In-repo encodings are
development-stage: ClickHouse demo tables are re-ingested; JSON encodings of
ids are strings of the raw `uint64` and re-derive. The one semantic migration
is `TagValue(0)` used as a registry base (keelson runtime vocabulary): the
registry base moves to a valid tag value as part of the port.

### SD9 — fibonaccicode hardening

Independent of the scheme switch: range-guard the width helpers (no negative
shifts for width ≥ 64), validate iterator inputs instead of panicking,
give `DecodeFibonacciCode` an explicit invalid result for comma-less input
(currently returns 2⁶⁴−1 silently), and rename/re-document the
`MaxFibonacciCodeRepresentable*` family to say what it computes — these are
exclusive bounds (counts of representable values), documented today as
inclusive maxima, which is precisely the confusion that produced the class-
bounds off-by-one. The width conventions (encode returns full width including
the comma bit; the comma finder returns one less) are documented at the
declarations. The OEIS-backed golden table and roundtrip tests stay.

## Alternatives considered

- **Keep fixed-width (status quo).** Retains the build-tag axis (one global
  width per binary, editor/CI coupling), a compose-time guard that cannot be
  made airtight without knowing per-id intent, and no frequency adaptivity.
  Its one operational defense — SQL-side decode cost — is now void. Rejected.
- **Ship both schemes.** Doubles the API surface and keeps the mixed
  convention seam alive — the review's worst defect cluster sits exactly on
  that seam today. A second scheme also reintroduces the compile-time or
  runtime mode the retirement is meant to remove. Rejected.
- **Dictionary-only ClickHouse decode.** Requires dictionary infrastructure
  on every query path, including ad-hoc `clickhouse-local` use; the pure-SQL
  decoder costs a dozen integer ops and no infrastructure. Dictionaries stay
  available later for id→name presentation. Rejected as the primary path.
- **Float `log2` for the MSB scan in SQL.** Precision cliffs near 2⁵³ make it
  unverifiable across the full domain; the or-shift smear plus `bitCount` is
  exact and uses only ancient built-ins. Rejected.
- **Widen `TagValue` to `uint64` with a width-cap policy.** Breaks the type
  for all consumers, weakens the ≥17-bit body floor, and no consumer needs
  tag values beyond `uint32`. Rejected; revisit only with a concrete need.

## Consequences

Positive: the `./tags` file loses its identifier entry (gopls/CI simplify);
ids are self-describing in Go, SQL, and any future reader; per-tag capacity
matches tag frequency; exhaustion and boundary behaviour become unit-testable;
~1,900 lines of unconsumed or scheme-dead code leave the tree; the nine
review defects are fixed or made structurally impossible.

Negative / accepted: splitting an id under an *unknown* tag costs ~12 integer
ops instead of one mask (known-tag operations are unchanged); ordering raw
ids across different tags follows code bit patterns, not tag-value order
(within a tag, ordering is unchanged — the property column compression relies
on); this is a breaking change for anything holding a `TagValue(0)` or the
removed constants, accepted under the clean break (SD8).

## Implementation slices

1. **SD9** fibonaccicode hardening — lands independently, no external breaks.
2. **SD1/SD2** identifier v2: variable-width core + Go split + property and
   golden tests; delete the build-tag files; sweep `./tags`, docs, consumers.
3. **SD6** identgen port and hardening (+ exhaustion, contention, chunked-
   batch tests).
4. **SD5** nanopass macro pack + emitted UDFs + Go↔SQL golden lock in CI.
5. **SD4** tag planning API + CLI advisor rebuild.
6. **SD7/SD8** culls, vocabulary registry base migration, demo re-ingest,
   documentation pass (identgen explanation rewritten; its stale
   external-repository provenance note is removed per the privacy rule).

## Validation

The decoding contract was validated ahead of this ADR: 2,817 golden ids
(Go encoder → CSV → `clickhouse-local` 26.6, decoder from SD2) with zero
mismatches on width, tag value, and body; widths 2–47 exercised, including
all fibonacci boundaries ±1 and maximal bodies. The same corpus-and-decoder
pair becomes the committed cross-language test in slice 4.

## Status

Accepted — 2026-07-05 (reviewed by @spx). Implemented so far:

- Slice 1 (SD9 hardening, plus the fibonacci-package input validation and
  the class-bounds off-by-one fix that SD4 will later subsume).
- Slice 2 (SD1–SD3): identifier v2 with fibonacci-coded tags, the
  `identifier_tag_fixed*` files and the `./tags` entry deleted, the runtime
  vocabulary registry base moved off tag value 0, and the tag-value registry
  rejecting invalid effective tag values. The full-repo build and test suite
  pass without the build tag; no goldens pinned composed id bit patterns, so
  the SD8 clean break required no data regeneration.
- Slice 3 (SD6): per-tag factory exclusivity (`identgen.ErrTagInUse`),
  chunked batch commits with one amendment to the SD6 text — a batch that
  overruns the tag's id space persists the mappings minted before the
  overrun instead of assigning nothing, because consumed sequence values
  cannot be returned and an empty abort would burn the space unmapped; the
  in-memory backend still pre-checks exactly and assigns nothing. Exhaustion
  is tested end to end through the 2^17-1-body widest-uint32 tags.

Slices 4–6 are pending. The SD2 decoder and the defect inventory in Context
were validated by an uncommitted review harness, to be recreated as committed
tests by slice 4.
